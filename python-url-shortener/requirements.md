# Requirements Document

## Introduction

This document defines the requirements for a Python URL Shortener service. The service accepts long URLs submitted by users and generates short, unique codes that redirect to the original URLs. The system exposes an HTTP API for creating short links, resolving them to their original destinations, and retrieving basic usage statistics. Statistics are returned only when explicitly requested by a client.

## Glossary

- **URL_Shortener**: The overall Python application providing URL shortening and redirection functionality.
- **Original_URL**: A valid, fully-qualified URL submitted by a user to be shortened.
- **Short_Code**: A unique, compact alphanumeric identifier generated by the system to represent an Original_URL.
- **Short_URL**: The full shortened URL composed of the service base URL and a Short_Code (e.g., `http://short.ly/abc123`).
- **Redirect**: The HTTP response that sends a client from a Short_URL to the Original_URL.
- **Code_Generator**: The component responsible for producing unique Short_Codes.
- **Store**: The persistence component responsible for storing and retrieving URL mappings.
- **API**: The HTTP interface through which clients interact with the URL_Shortener.
- **Client**: Any application or user agent that calls the API.
- **Request**: An HTTP call made by a Client to the API.
- **Mapping**: A stored association between a Short_Code and an Original_URL.
- **Access_Count**: The number of times a Short_Code has been resolved via a Redirect.

## Requirements

### Requirement 1: URL Shortening

**User Story:** As a Client, I want to submit a long URL and receive a Short_URL, so that I can share a compact link that redirects to the original destination.

#### Acceptance Criteria

1. WHEN a Client submits a valid Original_URL (with an http or https scheme and a well-formed host) that has not previously been shortened via a POST request to the `/shorten` endpoint, THE API SHALL create a new Mapping and return an HTTP 200 response containing the Short_URL.
2. WHEN a Client submits a valid Original_URL that has already been shortened, THE API SHALL return an HTTP 200 response with the existing Short_URL for that Original_URL without creating a new Mapping.
3. IF a Client submits a request body that does not contain a URL field, THEN THE API SHALL return an HTTP 422 response with an error message indicating that the URL field is required.
4. IF a Client submits a value that is not a valid URL (e.g., missing scheme, malformed host, unsupported scheme other than http or https), THEN THE API SHALL return an HTTP 422 response with an error message indicating the URL is invalid.
5. WHEN a new Mapping is created, THE Code_Generator SHALL produce a Short_Code composed solely of alphanumeric characters (A-Z, a-z, 0-9) with a length between 6 and 12 characters inclusive.
6. IF the Code_Generator produces a Short_Code that is outside the 6-12 character range or already exists in the Store, THEN THE Code_Generator SHALL regenerate a new Short_Code; IF the Code_Generator fails to produce a valid unique Short_Code after 10 attempts, THEN THE API SHALL return an HTTP 500 response with an error message indicating code generation failed.
7. THE Short_URL returned in the HTTP 200 response SHALL be composed of the service base URL followed by a forward slash and the Short_Code (e.g., `http://short.ly/abc123`).

### Requirement 2: URL Redirection

**User Story:** As a Client, I want to visit a Short_URL and be redirected to the original long URL, so that I arrive at the intended destination transparently.

#### Acceptance Criteria

1. WHEN a Client sends a GET request to `/{short_code}` with a Short_Code that exists in the Store and has not expired, THE API SHALL return an HTTP 302 redirect response with the `Location` header set to the Original_URL.
2. IF a Client sends a GET request to `/{short_code}` with a Short_Code that does not exist in the Store, THEN THE API SHALL return an HTTP 404 response with an error message indicating the Short_Code was not found.
3. WHEN the API processes a redirect for a Short_Code, THE Store SHALL increment the Access_Count for that Mapping by 1 before the redirect response is returned to the Client.
4. IF the Store is unreachable when a Client sends a GET request to `/{short_code}`, THEN THE API SHALL return an HTTP 503 response with an error message indicating the service is temporarily unavailable.
5. IF the Store increment of Access_Count fails after the redirect response has been determined, THE API SHALL still return the HTTP 302 redirect response and SHALL NOT block the redirect due to the count failure.

### Requirement 3: URL Mapping Persistence

**User Story:** As a Client, I want shortened URLs to remain valid across service restarts, so that links I share continue to work reliably over time.

#### Acceptance Criteria

1. THE Store SHALL persist all Mappings (Short_Code, Original_URL, creation timestamp, Access_Count) to durable storage.
2. WHEN the URL_Shortener starts, THE Store SHALL load all previously persisted Mappings before the API begins accepting requests, so that existing Short_URLs remain resolvable from the first request.
3. IF the durable storage is unreachable when the URL_Shortener starts, THEN THE URL_Shortener SHALL log an error and refuse to start, returning a non-zero exit code.
4. WHEN a new Mapping is created, THE Store SHALL persist it to durable storage before the API returns the Short_URL to the Client.
5. IF a write operation to durable storage fails (for either a new Mapping or an Access_Count update), THEN THE API SHALL return an HTTP 500 response with an error message indicating a storage failure and SHALL NOT return a Short_URL to the Client.

---

### Requirement 4: Link Statistics

**User Story:** As a Client, I want to retrieve usage statistics for a Short_URL, so that I can understand how often a link has been accessed.

#### Acceptance Criteria

1. WHEN a Client sends a GET request to `/stats/{short_code}` for an existing Short_Code, THE API SHALL return an HTTP 200 response with a JSON body containing the Short_Code, Original_URL, creation timestamp (in ISO 8601 format), and Access_Count.
2. IF a Client sends a GET request to `/stats/{short_code}` for a Short_Code that does not exist in the Store, THEN THE API SHALL return an HTTP 404 response with an error message indicating the Short_Code was not found.
3. THE API SHALL return the Access_Count reflecting all Redirect events that have completed at the time the `/stats/{short_code}` request is processed.
4. IF the Store is unreachable when a Client sends a GET request to `/stats/{short_code}`, THEN THE API SHALL return an HTTP 503 response with an error message indicating the service is temporarily unavailable.

---

### Requirement 5: Custom Short Codes

**User Story:** As a Client, I want to specify a custom Short_Code when shortening a URL, so that I can create memorable links for sharing.

#### Acceptance Criteria

1. WHERE a Client provides a custom Short_Code in the POST request to `/shorten`, THE API SHALL use that Short_Code to create the Mapping instead of generating one automatically and SHALL return the Short_URL containing the custom Short_Code in the HTTP 200 response.
2. WHERE a Client provides a custom Short_Code, THE API SHALL validate that the Short_Code contains only alphanumeric characters and hyphens and is between 3 and 50 characters in length.
3. IF a Client provides a custom Short_Code that already exists in the Store, regardless of whether it maps to the same or a different Original_URL, THEN THE API SHALL return an HTTP 409 response with an error message indicating the Short_Code is already in use.
4. IF a Client provides a custom Short_Code that fails validation, THEN THE API SHALL return an HTTP 422 response with an error message indicating which validation rule was violated.

---

### Requirement 6: URL Expiration

**User Story:** As a Client, I want to set an expiration time on a shortened URL, so that the link becomes inactive after a specified period.

#### Acceptance Criteria

1. WHERE a Client provides an expiration duration (in seconds, maximum 315,576,000 seconds) in the POST request to `/shorten`, THE Store SHALL record the expiration timestamp (creation time plus duration) alongside the Mapping, and THE API SHALL include the expiration timestamp in the HTTP 200 response.
2. WHEN a Client sends a GET request to `/{short_code}` for a Mapping whose expiration timestamp has passed (current time is strictly after the expiration timestamp), THE API SHALL return an HTTP 410 response with a message indicating the link has expired.
3. WHEN a Client sends a GET request to `/stats/{short_code}` for an expired Mapping, THE API SHALL return an HTTP 200 response with the statistics including the expiration timestamp without returning an error.
4. IF a Client provides an expiration duration that is not a positive integer or exceeds 315,576,000 seconds, THEN THE API SHALL return an HTTP 422 response with an error message indicating the expiration duration is invalid.
5. WHEN the API returns an HTTP 410 response for an expired Short_Code, THE Store SHALL NOT increment the Access_Count for that Mapping.

---

### Requirement 7: Health and Operational Readiness

**User Story:** As an operator, I want a health check endpoint, so that I can monitor whether the service and its dependencies are operational.

#### Acceptance Criteria

1. WHEN a Client sends a GET request to `/health` and the Store probe completes successfully within 1 second, THE API SHALL return an HTTP 200 response with a JSON body containing `{"status": "operational", "store": "reachable"}`.
2. IF the Store probe fails or does not complete within 1 second when a Client sends a GET request to `/health`, THEN THE API SHALL return an HTTP 503 response with a JSON body containing `{"status": "degraded", "store": "unreachable"}`.
3. THE API SHALL send the complete `/health` response to the Client within 2 seconds of receiving the request, measured from request receipt to response sent, inclusive of the Store probe.
